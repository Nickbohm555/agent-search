Application Reference

One document for how agent-search is built and why it behaves the way it does.

agent-search is a LangGraph-first, citation-grounded RAG application that exists in two forms at once: an SDK for direct runtime use and a containerized web application with a FastAPI backend, React/Vite frontend, pgvector-backed storage, canonical thread_id handling, and async lifecycle delivery for long-running runs and wiki ingestion.

Architecture

The core architectural pattern is a containerized service app wrapped around a reusable SDK runtime. The HTTP API is thin and mostly delegates to SDK entrypoints, while the actual question-answering logic now executes through a compiled LangGraph StateGraph over a canonical runtime state contract.

Primary Layers

  • Frontend UI in src/frontend/ for wiki loading, run submission, polling, and visualization.
  • FastAPI backend in src/backend/ for HTTP routing, dependency setup, and persistence.
  • SDK/runtime core in src/backend/agent_search/ for public API methods, LangGraph execution, checkpoint bootstrap, and runtime jobs.
  • Runtime nodes and compatibility services cooperate: LangGraph nodes live under agent_search/runtime/, while legacy service functions still supply node logic and mapping helpers.
  • Postgres + pgvector persistence stores internal documents and chunks, and Postgres also tracks runtime executions, checkpoint links, and idempotency effects.

Core Patterns

  • Entrypoint to router to service separation.
  • SDK as the main business-logic surface, with HTTP as a wrapper around it.
  • LangGraph state-graph execution with typed reducers, stable channels, and per-stage snapshots.
  • Stable thread_id identity across sync, async, retry, and resume flows.
  • Lifecycle events streamed over SSE while job status remains queryable over REST.
  • Environment-composed runtime via Docker Compose and env vars.
  • Fallback-first behavior when retrieval or model dependencies are unavailable or slow.

Key Architectural Decision

The repo keeps the runtime logic inside the Python package and reuses it from both the SDK-facing API and the web backend. That is the most important design choice in this codebase: the web app is not a separate implementation of the algorithm, it is a transport and orchestration layer over the same LangGraph runtime core. Phase 5 documents the project after the migration rather than describing a future cutover.

How It Works

The application has two major workflows: internal wiki ingestion and LangGraph runtime query execution. Query execution now has one canonical state-graph path, with sync, async, SSE, and resume behavior layered around that shared runtime.

1. Wiki Ingestion Flow

  1. The UI calls /api/internal-data/wiki-sources to discover curated load options and current loaded state.
  2. The user starts a sync or async load via /api/internal-data/load or /api/internal-data/load-async.
  3. services/internal_data_service.py resolves the wiki source, fetches documents, chunks them, embeds them, and writes vectors to the configured collection.
  4. Metadata rows are inserted into internal_documents, while chunk vectors live in the vector store layer.
  5. The frontend polls load status every second and can cancel an in-flight load job.

2. Agent Query Flow

  1. The UI submits a query through /api/agents/run-async and begins polling /api/agents/run-status/{job_id}.
  2. routers/agent.py builds runtime dependencies: a vector store pointed at the configured collection and a ChatOpenAI model instance.
  3. The router delegates to SDK functions in agent_search/public_api.py.
  4. The SDK validates inputs, normalizes config, preserves or mints a canonical thread_id, and sends the request into the LangGraph runtime.
  5. agent_search/runtime/graph/builder.py compiles the runtime graph and runtime/graph/execution.py executes it with LangGraph stream modes for values, tasks, updates, and checkpoints.
  6. The graph decomposes the query into sub-questions, expands queries, searches, reranks, answers each sub-question, and synthesizes a final answer into canonical RAGState.
  7. Stage snapshots and lifecycle events are emitted during the async run so the frontend can display the timeline, sub-questions, artifacts, retrieval counts, rerank bypasses, and citation coverage.

3. Runtime State And Recovery

  1. Canonical state: RAGState is the named runtime contract; LangGraph uses RuntimeGraphState to add graph-only channels such as initial search context.
  2. Stable identity: thread_id is validated as a UUID and stays stable across run creation, status reads, and resume operations.
  3. Checkpoint support: the durable runner can compile with a Postgres-backed LangGraph checkpointer, bootstrap checkpoint tables automatically, and resume from interrupt payloads.
  4. Execution ledger: runtime execution rows, checkpoint links, and idempotency effects are stored in Postgres so retries and resumptions have a durable record.
  5. Delivery split: REST status and SSE events are caller-facing, while checkpoints and execution identity are runtime-facing infrastructure.

4. Runtime Stages

  1. Decompose: turn the user question into narrow atomic sub-questions.
  2. Expand: generate alternate search queries for each sub-question.
  3. Search: retrieve candidate passages from the vector store.
  4. Rerank: reorder passages, usually with OpenAI-backed reranking.
  5. Answer: generate grounded sub-answers and verify whether each answer is supported.
  6. Synthesize: produce the final response and map citation indices back to stored evidence rows.

Fallback Behavior

This app is intentionally defensive. The service layer contains timeout guards and fallback text for retrieval outages, slow generations, missing API keys, malformed model outputs, rerank failures, and verification timeouts. The result is a system that often returns partial output rather than hard-failing.

API Surface

Agent Endpoints

  • POST /api/agents/run: synchronous run returning a RuntimeAgentRunResponse.
  • POST /api/agents/run-async: start async run and return job metadata.
  • GET /api/agents/run-status/{job_id}: retrieve status, stage snapshots, artifacts, partial outputs, and final result when available.
  • GET /api/agents/run-events/{job_id}: stream lifecycle events over server-sent events.
  • POST /api/agents/run-cancel/{job_id}: request cancellation.
  • POST /api/agents/run-resume/{job_id}: resume a paused run from checkpoint state.

Internal Data Endpoints

  • GET /api/internal-data/wiki-sources: curated wiki source list with already-loaded flags.
  • POST /api/internal-data/load: synchronous wiki ingestion.
  • POST /api/internal-data/load-async: start async ingestion.
  • GET /api/internal-data/load-status/{job_id}: poll ingestion progress.
  • POST /api/internal-data/load-cancel/{job_id}: cancel ingestion.
  • POST /api/internal-data/wipe: delete internal documents and chunks.

Public SDK Surface

The public Python entrypoints live in src/backend/agent_search/public_api.py. The main methods are advanced_rag(), run() as a deprecated alias, run_async(), get_run_status(), resume_run(), and cancel_run().

Async Contract

Async runs still execute through an in-process ThreadPoolExecutor and active job registry, but the runtime now also persists execution identity, checkpoint linkage, and idempotency records in Postgres. That means live status/SSE delivery remains single-process today, while run lineage and checkpoint recovery data survive independently of the in-memory job object.

Data Model

Persistent Tables

  • internal_documents: source metadata and original content for loaded wiki documents.
  • internal_document_chunks: per-chunk text, vector embedding, and JSON metadata.
  • runtime_execution_runs: canonical run and thread_id registry.
  • runtime_checkpoint_links: latest checkpoint linkage for resumable runs.
  • runtime_idempotency_effects: replay-safe effect ledger for durable execution paths.

Runtime State

  • RuntimeAgentRunRequest and RuntimeAgentRunResponse define the outer API contract.
  • RAGState is the canonical runtime state contract for decomposition results, artifacts, final answer text, citations, run metadata, and snapshots.
  • RuntimeGraphState wraps RAGState with LangGraph-only channels and reducer semantics.
  • SubQuestionArtifacts retains the evidence chain for each lane: expanded queries, retrieved docs, reranked docs, and citation rows.

Citation Mapping

Final citations are not generic references; they are rows mapped back to retrieved or reranked documents. Citation quality depends on chunk metadata such as title, source URL, and optional document identifiers being present at ingestion time.

Integrations

LLM And Reranking

  • OpenAI is the dominant provider in the current implementation.
  • Model usage appears in decomposition, query expansion, reranking, sub-answer generation, and synthesis.
  • Several paths degrade gracefully if the key is missing.

Tracing

  • Langfuse support is built into the SDK and runtime through callback handlers and trace/span metadata.
  • Tracing is optional and can be configured through settings or environment variables.

Source Systems

  • Wikipedia-derived curated sources are the primary built-in internal data source.
  • The ingestion path resolves wiki documents, chunks them, embeds them, and stores them locally.

Persistence

  • Postgres is the system of record.
  • pgvector supports semantic retrieval for internal document chunks.

Conventions

The broader planning notes had convention examples drawn heavily from other repos, but the conventions actually visible in this repo are straightforward and consistent.

Naming

  • Backend modules use snake_case.py.
  • Tests follow test_*.py on the backend and *.test.tsx on the frontend.
  • React components use PascalCase.tsx such as App.tsx.
  • Schema and model classes use PascalCase.

Code Shape

  • Routers stay thin and build dependencies, then delegate.
  • Services hold the real behavior and fallback rules.
  • Pydantic models define request, response, and graph-state contracts.
  • Logging is explicit and operationally oriented, especially around runtime stages and fallbacks.

Error Handling Style

Errors are either translated into HTTPException at API boundaries or mapped into SDK-specific error types. Internally, many functions prefer controlled fallback output over fatal exceptions, especially when LLM-dependent steps fail.

Concerns

Known Bug

The planning concerns correctly identify a CORS mismatch in src/backend/main.py: allow_origins=["*"] is combined with allow_credentials=True. That combination is invalid for credentialed browser requests and should be replaced with explicit origins.

Security Gap

The same wildcard CORS configuration is also the main documented security concern for this app. The application is otherwise lightweight and does not currently implement a deeper auth model.

Operational Fragility

  • Active async status and SSE delivery still depend on in-memory job objects, so process restarts can interrupt live polling behavior.
  • The backend resolves critical dependencies at request time rather than through a managed service container abstraction.
  • Many stages depend on OpenAI availability for best quality, even though fallback behavior exists.

Performance Limits

  • Multi-stage LLM execution can be slow, which is why the UI defaults to async polling and long frontend timeouts.
  • Each sub-question can fan out into multiple expanded queries, increasing search and rerank load.
  • The default threadpool-based async model is suited to local or small deployments, not large distributed workloads.

Tradeoffs

Why The Current Design Is Good

  • One LangGraph runtime implementation serves both SDK and web use cases.
  • Stage snapshots make the system inspectable, not just opaque.
  • Stable thread_id and checkpoint-aware execution improve operational traceability.
  • Curated wiki ingestion gives the app a deterministic internal corpus for demos and controlled tests.
  • Fallback logic prevents many empty-screen failures.

What The Design Trades Away

  • In-memory async job tracking still limits horizontal scaling even though run identity and checkpoints are persisted.
  • Router-level dependency construction is simple but couples request handling to concrete provider setup.
  • Heavy fallback behavior improves uptime but can mask quality regressions unless carefully monitored.
  • One main frontend page keeps the UI simple, but it concentrates a lot of workflow complexity in App.tsx.

Practical Reading Of The System

This is not just a CRUD app with an LLM call attached. It is a staged evidence-processing pipeline with an inspectable UI layered on top. The most important operational fact is that the system is designed to preserve intermediate reasoning artifacts and surface them back to the caller.

Operational Notes

  • The backend container runs migrations on startup before launching Uvicorn.
  • The frontend expects the backend base URL via VITE_API_BASE_URL and uses long request timeouts for runs and ingestion.
  • The backend logs runtime progress aggressively to stdout to make container debugging easier.
  • The default vector collection name is agent_search_internal_data.
  • Initial search context retrieval is currently disabled in the runtime runner; the LangGraph path proceeds with an empty initial context before decomposition.
  • The migration to LangGraph-first execution is complete; remaining release work is documentation, SDK surface hygiene, and migration guidance rather than runtime cutover.
