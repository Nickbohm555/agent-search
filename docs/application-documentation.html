Workspace Map

Architecture

The workspace does not enforce one shared runtime architecture. Instead, each project root owns a distinct execution style while following a common preference: thin entrypoints, explicit boundaries, and infrastructure composed through environment variables and container orchestration.

Dominant Patterns

Containerized Service Apps

agent-search and agent-trace split frontend, backend, and database services and run them through Docker Compose.

  • FastAPI backend
  • React/Vite frontend
  • PostgreSQL-backed persistence
  • Router to service separation

Pipeline CLI

cli-skill models workflow as explicit stages with approval gates and deterministic package boundaries.

  • Command entrypoint via Cobra
  • Stage-based package composition
  • Installation transactions
  • Controlled crawl and content refinement

Platform Monolith

onyx is large and modular, but operationally behaves as a central platform backend plus a separate Next.js frontend.

  • Large FastAPI app factory
  • Feature-oriented server modules
  • App-router web surface
  • Broad infrastructure footprint

Workflow Layer

dream-workflow and multiple AGENTS.md files define the process layer that guides planning, execution, and repeated work loops.

  • Prompt-driven flow
  • Shell loop automation
  • Operational constraints and runbooks

Key Abstractions

Router-Service Boundary

HTTP handlers remain thin and delegate business logic to services or domain modules.

Graph And Pipeline Runtime

AI workflows are modeled as staged or graph-like execution units with typed inputs and explicit sequencing.

Environment-Composed Runtime

Infrastructure dependencies are configured through compose files and environment variables, not a single in-process binary.

Data Flows

The core application behaviors can be understood through a small set of repeated runtime flows.

Agent Search Request Flow

  1. The frontend mounts from agent-search/src/frontend/src/main.tsx and calls the backend using configured API URLs.
  2. agent-search/src/backend/main.py registers the FastAPI app and routes requests into routers/agent.py.
  3. The router delegates to services/agent_service.py and runtime orchestration in agent_search/runtime/runner.py.
  4. Typed schema objects under schemas/ shape request and response data.

Trace Analysis Flow

  1. /api/tracer/run enters agent-trace/src/backend/routers/tracer.py.
  2. The router composes TraceAnalyzerService and invokes orchestration in services/trace_analyzer_service.py.
  3. The service fetches or persists traces, prepares sandbox context, runs the tracer graph in agents/deep_agent_tracer.py, and aggregates change sets.
  4. Responses are returned via typed API schema models such as schemas/tracer_api.py.

Onyx Platform Flow

  1. The backend starts in onyx/backend/onyx/main.py through get_application() and lifespan initialization.
  2. The main application includes many feature routers from server and feature packages.
  3. The web application under onyx/web/src/app/ interacts with those APIs using Next.js routing and rewrite configuration.
  4. Persistent and background state is distributed across Postgres, Redis, search/index services, object storage, and model-serving infrastructure.

Technology Stack

Primary Languages

  • Python 3.11+ for backend services and platform APIs.
  • TypeScript for React/Vite and Next.js frontends.
  • Go 1.25.x for CLI workflow tooling.
  • Rust in the Tauri desktop shell inside Onyx.
  • YAML for Docker Compose and deployment wiring.

Runtime And Package Managers

  • Python projects use pyproject.toml and a uv/PEP 621 workflow.
  • Frontend projects use npm.
  • CLI code uses Go Modules.
  • Docker Compose is the main local orchestration layer for service applications.

Frameworks

  • FastAPI for APIs.
  • React + Vite for lightweight service-app frontends.
  • Next.js for the Onyx web application.
  • Cobra for the Go CLI command system.

Core Infrastructure Libraries

  • sqlalchemy, psycopg, and pgvector for relational/vector data.
  • openai, LangChain packages, and langfuse for LLM-related execution and tracing.
  • redis and celery in Onyx for cache and background execution.
  • OpenSearch/Vespa-related dependencies for search and indexing in Onyx.

Configuration Model

Environment variables are the dominant configuration contract. Service apps rely on DATABASE_URL, Postgres credentials, frontend API base URLs, and provider keys. Onyx expands this substantially with auth mode, Redis, search/index hosts, model server settings, and S3-compatible storage configuration.

Development Requirements

Docker + Docker Compose Python 3.11+ Node.js + npm Go 1.25.x

Integrations

LLM Providers

  • OpenAI is a primary inference provider across projects.
  • Anthropic is supported as an optional provider in the agent apps and Onyx-related flows.
  • Provider credentials are environment-driven, primarily via OPENAI_API_KEY and ANTHROPIC_API_KEY.

Observability

  • Langfuse supports tracing and runtime observability across agent-oriented systems.
  • Sentry is present for Onyx backend and web error tracking.

Data And State Systems

  • PostgreSQL is the primary relational store.
  • Agent apps add pgvector for vector-backed workflows.
  • Onyx uses Redis for cache and coordination.
  • Onyx uses S3-compatible object storage with MinIO defaults in local deployment.

External SaaS And Connectors

Onyx integrates with a broad connector set, including Slack, Stripe, SendGrid, GitHub, Jira, Salesforce, Asana, Dropbox, and additional third-party systems.

Authentication

Authentication strategy differs by project. The two smaller agent services are largely API-focused and environment-configured, while Onyx supports multiple auth modes, including OAuth and SAML-related flows with callback routes in the web application.

Deployment Model

Docker Compose is the primary local and self-host deployment model for the service apps and for the Onyx stack. Onyx also supports registry-based image deployment through compose-defined image references.

Conventions

Naming

  • Python modules: snake_case.py
  • Python tests: test_*.py
  • React components: PascalCase.tsx
  • Frontend logic and hooks: camelCase.ts
  • Next.js routes: page.tsx per route segment
  • Go source: snake_case.go with paired _test.go

Code Style

  • Web formatting uses Prettier.
  • Onyx backend uses Ruff with target py311 and line length 130.
  • Backend code is expected to be type-heavy and mypy-clean.
  • Use descriptive names and explicit typed parameters.

Imports

  1. External packages first.
  2. Internal aliases second.
  3. Relative imports last.

Onyx web uses aliases such as @/*, @tests/*, @public/*, and @opal/*.

Error Handling And Logging

  • Frontend fetch flows use typed errors and propagate status/detail information.
  • Backend handlers translate domain failures into explicit HTTPException responses.
  • Python logging uses centralized logger setup instead of prints.
  • Comments are used for rationale, non-obvious setup, and operational tradeoffs.

Testing

Testing expectations are strongest in the Onyx repository, where the planning docs identify distinct unit, integration, external dependency unit, daily, and browser-level end-to-end layers.

Backend

  • Pytest is the primary runner.
  • Tests live under onyx/backend/tests/ with unit, integration, and daily segregation.
  • Mocks are used in isolated unit tests; integration tests rely on running deployment infrastructure.

Web

  • Jest and Testing Library support unit and component integration testing.
  • Tests are primarily colocated under onyx/web/src/**.
  • Shared setup utilities live under onyx/web/tests/setup.

E2E

  • Playwright covers browser workflows under onyx/web/tests/e2e.
  • Some flows remain flaky or skipped, leaving known gaps in drag-and-drop and similar UX behavior.

Typical Commands

  • cd onyx/web && npm test
  • cd onyx/web && npm run test:coverage
  • source .venv/bin/activate && cd onyx && pytest -xv backend/tests/unit
  • cd onyx/web && npx playwright test <TEST_NAME>

Testing Patterns

  • Frontend tests favor user-visible assertions and explicit API call expectations.
  • Backend unit tests patch DAL and service boundaries.
  • Shared fixtures and test utilities centralize repetitive setup.
  • No global backend coverage threshold is currently enforced.

Risks And Concerns

Known Bugs

  • agent-search has a CORS configuration mismatch: wildcard origins with credentials can break browser requests and is also a security concern.
  • Onyx has a skipped flaky E2E path around assistant drag-and-drop ordering.

Security

  • Wildcard CORS with credentials in agent-search should be replaced with explicit trusted origins.
  • Per-user MCP permissions in Onyx are explicitly incomplete.

Tech Debt

  • onyx/backend/onyx/main.py is an oversized startup and route-composition hotspot.
  • Deprecated or transitional schema contracts remain active in some Onyx manage models.

Performance

  • Onyx upload paths read full payloads into memory.
  • cli-skill may duplicate fetch pressure by re-fetching content across crawl and process stages.

Fragility

  • Onyx process lifecycle management has acknowledged IO fragility in sandbox local process management.
  • Onyx startup couples auth, infra, metrics, and router assembly into one broad blast-radius path.

Scaling Limits

  • cli-skill defaults to a 50-page crawl cap, limiting large documentation coverage.
  • Onyx connector orchestration remains highly dependent on Redis availability.

Coverage Gaps

The current planning audit explicitly calls out under-tested or unstable behavior in assistant ordering, scroll behavior, web-search providers, and selected connector integration flows. This should be treated as active risk when changing those surfaces.

Operational Guidance

Where To Add Backend Features

  • agent-search/src/backend/services/ and routers/ for agent-search backend work.
  • agent-trace/src/backend/services/ and routers/ for trace-analysis backend work.
  • cli-skill/internal/<domain>/ for CLI functionality.
  • onyx/backend/onyx/server/features/<feature>/ for Onyx feature endpoints.

Where To Add Frontend Or Web Features

  • agent-search/src/frontend/src/ and agent-trace/src/frontend/src/ for service-app UI work.
  • onyx/web/src/app/ and onyx/web/src/components/ for Onyx web work.

Where To Add Tests

  • agent-search/src/backend/tests/
  • agent-trace/src/backend/tests/
  • cli-skill/internal/**/*_test.go
  • onyx/backend/tests/ and onyx/web/tests/

Generated Or Special Directories

  • .planning/codebase/ contains generated planning artifacts and is committed.
  • onyx/backend/generated/ contains generated backend artifacts and is committed.
  • Frontend node_modules/ trees are generated and not committed.
