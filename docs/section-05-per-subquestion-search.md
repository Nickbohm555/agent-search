# Section 5 Architecture: Per-subquestion search

## Purpose
Run retrieval for each decomposed sub-question using the expanded query from Section 4 (or the raw sub-question as fallback), and preserve the ranked retrieval output per sub-question for downstream stages.

## Components
- Retriever tool API: `search_database(...)` in `src/backend/tools/retriever_tool.py`.
- Coordinator subagent contract: `_RAG_SUBAGENT_PROMPT` in `src/backend/agents/coordinator.py` (requires both `query` and `expanded_query`).
- Search callback capture: `SearchDatabaseCaptureCallback` in `src/backend/utils/agent_callbacks.py`.
- Runtime extraction and shaping:
- `_extract_sub_qa(...)` in `src/backend/services/agent_service.py`.
- `_estimate_retrieved_doc_count(...)` in `src/backend/services/agent_service.py`.
- Data model carrying search outputs: `SubQuestionAnswer` in `src/backend/schemas/agent.py`.
- Runtime orchestrator entrypoint: `run_runtime_agent(...)` in `src/backend/services/agent_service.py`.

## Data Flow
### Inputs
- Sub-question units delegated by coordinator `task()` calls.
- Retriever tool-call payload fields:
- `query`: exact sub-question text.
- `expanded_query`: expansion generated by subagent (optional but expected by prompt).
- Retrieval config in tool call:
- `limit` (default `10`).
- optional `wiki_source_filter`.
- Vector-store indexed chunks and metadata.

### Transformations and movement
1. Coordinator delegates each sub-question to `rag_retriever`.
2. Subagent calls `search_database(query, expanded_query, limit, wiki_source_filter)`.
3. Retriever computes effective lookup text:
- `retrieval_query = (expanded_query or "").strip() or query`
- This ensures retrieval still runs when expansion is empty.
4. Retriever performs semantic search:
- `vector_store.similarity_search(retrieval_query, k=safe_limit, filter=filter_payload)`.
5. Retriever formats ranked output into numbered lines:
- `1. title=... source=... content=...`
- `2. ...`
6. `SearchDatabaseCaptureCallback` intercepts each `search_database` run and stores ordered `(input_str, output_str)` pairs.
7. After agent execution, `run_runtime_agent(...)` pulls captured calls and passes them to `_extract_sub_qa(...)`.
8. `_extract_sub_qa(...)` maps each captured search call into `SubQuestionAnswer`:
- `sub_question` from tool input payload,
- `expanded_query` parsed from payload,
- `sub_answer` set to ranked retrieval output string,
- `tool_call_input` preserved for observability,
- `sub_agent_response` paired from delegated task final answer when available.
9. Runtime logs per item include `sub_question`, `expanded_query`, and `docs_retrieved` computed by `_estimate_retrieved_doc_count(...)`.
10. Resulting `sub_qa` list is handed to Section 6 (document validation) as the direct retrieval artifact.

### Outputs
- Per-sub-question ranked retrieval payload stored in `SubQuestionAnswer.sub_answer`.
- Preserved `expanded_query` and raw tool input for traceability/debugging.
- Search-stage observability logs:
- callback capture count,
- per-subquestion retrieval count estimates,
- effective retrieval query details in retriever logs.

### Data boundaries
- Boundary A: coordinator/subagent LLM decision -> typed retriever tool invocation.
- Boundary B: retriever output text -> callback capture buffer (`_calls` list).
- Boundary C: callback tuples -> typed `SubQuestionAnswer` objects in service layer.
- Boundary D: `SubQuestionAnswer` search artifacts -> validation/reranking pipeline stages.

## Key Interfaces and APIs
- `search_database(query: str, expanded_query: str | None = None, limit: int = 10, wiki_source_filter: str | None = None) -> str`
- `SearchDatabaseCaptureCallback.get_calls() -> list[tuple[str, str]]`
- `_extract_sub_qa(messages, search_database_calls=None) -> list[SubQuestionAnswer]`
- `SubQuestionAnswer` fields used by this section:
- `sub_question`
- `sub_answer` (ranked retrieval rows)
- `tool_call_input`
- `expanded_query`
- `run_runtime_agent(payload, db) -> RuntimeAgentRunResponse` (search capture + extraction wiring)

## Fit With Adjacent Sections
- Upstream (Section 4): consumes `expanded_query` produced during sub-question handling.
- Downstream (Section 6): provides raw ranked retrieval rows that validation filters.
- Downstream (Section 7): ranking quality depends on recall/precision established here.
- Downstream (Sections 8-9): subanswer generation and verification quality is constrained by retrieval quality from this section.

## Tradeoffs
### Chosen design
Use a tool-based retrieval path inside the subagent and reconstruct per-subquestion search artifacts via callback-captured tool input/output pairs.

### Benefits
- Strong observability: exact tool input/output is preserved and logged.
- Minimal new infrastructure: reuses existing vector store and agent tool plumbing.
- Deterministic fallback: empty `expanded_query` safely falls back to `query`.
- Easy handoff: output shape already matches downstream `SubQuestionAnswer` pipeline expectations.

### Costs
- Retrieval output is text-formatted rows, not structured typed documents, so later stages must parse strings.
- Callback-order coupling assumes retrieval calls align with delegated subquestion ordering.
- Tool-call text parsing can be brittle if payload formatting changes.

### Alternatives considered or rejected
- Direct service-layer retrieval per subquestion (no tool-call capture):
- Pros: typed outputs, fewer parse steps, clearer contracts.
- Cons: less transparency into actual agent tool behavior; diverges from agent-native execution path.
- Return structured JSON from retriever tool instead of formatted text:
- Pros: safer downstream parsing, less regex/string dependence.
- Cons: larger payloads in message history, extra formatting complexity in agent prompts/responses.
- Persist retrieval artifacts in a dedicated runtime store/table:
- Pros: stronger durability and replay/debug capabilities.
- Cons: more schema/storage complexity and higher write overhead for each run.
