# Section 4 Architecture: Per-Subquestion Query Expansion

## Purpose
Produce an expanded retrieval query for each delegated atomic sub-question so retrieval can use synonym/reformulation signals without changing the user-visible sub-question text.

## Data Flow
Inputs:
- `sub_question` delegated by coordinator `task()` call.
- Optional `expanded_query` generated by subagent prompt rules.
- Optional retrieval controls (`limit`, `wiki_source_filter`).

Transformations:
1. Subagent prompt enforces dual-field tool input: exact `query` plus derived `expanded_query`.
2. Retriever computes `retrieval_query = expanded_query (trimmed) || query`.
3. Vector store search executes only on `retrieval_query`; results are normalized to numbered text rows.
4. `_extract_sub_qa(...)` parses tool call payload to recover both `sub_question` and `expanded_query`, then stores them in `SubQuestionAnswer`.
5. Later per-subquestion reranking uses `expanded_query` first (`item.expanded_query.strip() or item.sub_question`) so expansion continues influencing ranking.

Outputs:
- Retrieval output text (ranked rows) tied to each sub-question.
- `SubQuestionAnswer.expanded_query` persisted in runtime response and logs.

Data movement and boundaries:
- Agent boundary: natural-language prompt -> tool-call JSON args (`query`, `expanded_query`).
- Tool boundary: JSON args -> vector-store similarity query string.
- Service boundary: raw agent/tool messages -> typed `SubQuestionAnswer` models.

## Key Interfaces / APIs
- `search_database(query: str, expanded_query: str | None = None, limit: int = 10, wiki_source_filter: str | None = None) -> str`
- `_parse_tool_input_for_expanded_query(input_str: str) -> str`
- `_extract_sub_qa(messages, search_database_calls=None) -> list[SubQuestionAnswer]`
- `SubQuestionAnswer.expanded_query: str = ""`
- RAG subagent contract in `_RAG_SUBAGENT_PROMPT`:
- always pass both `query` and `expanded_query`
- fallback to `expanded_query == query` when no useful expansion exists

## How It Fits Adjacent Sections
- Upstream (Section 3): consumes decomposition output (atomic `sub_question` strings).
- Immediate downstream (Section 5): retrieval execution is where expansion first changes behavior by selecting `retrieval_query`.
- Later downstream (Sections 6-9): validation/reranking/answering/verification operate on retrieved artifacts; reranking still reuses `expanded_query`, so Section 4 affects more than first-pass retrieval.

## Tradeoffs
1. Prompt-generated expansion vs deterministic synonym rules
- Chosen: expansion is generated by the subagent prompt.
- Pros: flexible across domains and phrasing; no synonym dictionary maintenance.
- Cons: non-deterministic phrasing can vary by run/model behavior.
- Rejected alternative: static keyword expansion tables.
- Why rejected: predictable but brittle and expensive to keep current.

2. Single effective retrieval query vs multi-query union retrieval
- Chosen: retriever executes one `retrieval_query` (expanded preferred, fallback original).
- Pros: simple interface, bounded latency/cost, easy logging.
- Cons: may miss documents that one-query union would catch.
- Rejected alternative: run both original and expanded queries then merge.
- Why rejected: higher compute and dedup complexity for this iteration.

3. Keep `query` and `expanded_query` both in payload vs overwrite original query
- Chosen: preserve both fields.
- Pros: auditability and debugging (compare intent vs expansion), safer downstream interpretation.
- Cons: larger tool payloads and more parsing logic.
- Rejected alternative: mutate `query` in place and send one field.
- Why rejected: loses provenance and makes behavior harder to inspect.

4. Parse tool input from serialized strings vs introducing stricter typed event channel
- Chosen: tolerant parsing (`json.loads` then `ast.literal_eval`) from existing message payloads.
- Pros: works with current deep-agent message shapes and callback capture paths.
- Cons: extra parsing complexity and potential edge cases with malformed payloads.
- Rejected alternative: dedicated typed runtime event object for tool calls.
- Why rejected: larger refactor beyond this section scope.
