Run Button Flow: End-to-End Data Path

This document traces exactly what runs when you click Run in the UI, from frontend submit to backend completion and response rendering.

High-Level Path

Frontend: App.tsx::handleRun API client: startAgentRun() + getAgentRunStatus() Backend route: POST /api/agents/run Async route: POST /api/agents/run-async Status route: GET /api/agents/run-status/{job_id} Service: run_runtime_agent() Pipeline: graph runner (decompose -> expand -> search -> rerank -> answer -> synthesize) Response model: RuntimeAgentRunResponse

Graph State Contracts

Runtime graph state is typed in backend schemas as AgentGraphState with staged SubQuestionArtifacts and compatibility fields (sub_qa, output). Node IO is defined via stable contracts: decompose, expand, search, rerank, answer_subquestion, and synthesize_final.

Observability identifiers are carried in GraphRunMetadata: run_id, thread_id, trace_id, correlation_id.

The runtime path is graph-only: run_runtime_agent(...) calls run_parallel_graph_runner(...) for /api/agents/run. run_sequential_graph_runner(...) is retained as a deterministic reference runner for tests.

run_parallel_graph_runner(...) executes bounded fanout across sub-questions with deterministic output reindexing, plus stage_snapshots emitted on graph state after each stage for progressive status consumers.

Async polling contracts in services/agent_jobs.py and routes: /api/agents/run-async, /api/agents/run-status/{job_id}, /api/agents/run-cancel/{job_id}. Status emits stage subquestions_ready as soon as decomposition snapshot is produced.

The frontend Search stage panel consumes sub_question_artifacts[].retrieved_docs and sub_question_artifacts[].retrieval_provenance to display merged candidate previews plus transparent merge stats (raw_hits vs deduped_hits) before rerank.

The frontend Rerank stage panel consumes sub_question_artifacts[].reranked_docs to display final evidence order, citation indices, optional scores, and a visible fallback badge when reranking is bypassed.

The frontend Subanswer stage panel consumes sub_qa[].sub_answer, surfaces explicit fallback nothing relevant found, and links subanswer citation markers (for example [1]) back to reranked evidence rows.

The frontend Final Synthesis panel updates only after terminal synthesize_final completion, summarizes supporting subanswers and citation coverage, and preserves the previous successful final answer while a new run is still in progress.

Backend regression tests verify stable graph-runtime response shape (main_question, sub_qa, output) and fallback behavior (for example vector-store timeout and nothing relevant found behavior).

Retrieval-quality evaluations focus on evidence quality only (not cost): tests compare search-only vs search+rerank for hard queries, track top-hit relevance and citation-grounding consistency, and include baseline slices against the runtime stack (MultiQueryRetriever expansion + flashrank rerank).

Efficiency evaluations focus on token/cost impact: tests compare naive large-context retrieval windows against reranked top_n context, require that gold evidence remains in the final cited context, and record operating targets of k_fetch=6..8 with top_n=2..3 for strong token-budget reduction.

Where Parallelism Happens

  • run_pipeline_for_subquestions uses ThreadPoolExecutor with max_workers = min(SUBQUESTION_PIPELINE_MAX_WORKERS, len(sub_qa)).
  • Each worker runs _run_pipeline_for_single_subquestion independently.
  • Inside each worker, document validation calls validate_subquestion_documents, which itself parallelizes per-document validation using another ThreadPoolExecutor.
  • Refinement branch retrieval (_seed_refined_sub_qa_from_retrieval) also runs in parallel across refined subquestions.
  • Order is preserved after parallel execution by writing each future result back to its original index before returning the final list.
Parallel by subquestion Parallel by document validation Nested thread pools can increase concurrency load

Subquestion Worker Logic (Exact)

_run_pipeline_for_single_subquestion(item): working_item = deep copy of item # 1) Document validation working_item.sub_answer = validate/filter retrieved docs # 2) Reranking rerank_query = expanded_query or sub_question working_item.sub_answer = reranked citation-contract lines reranked_output = working_item.sub_answer # snapshot for verification evidence # 3) Subanswer generation working_item.sub_answer = generate_subanswer(sub_question, reranked_output) # 4) Verification verification = verify_subanswer(sub_question, sub_answer, reranked_output) working_item.answerable = verification.answerable working_item.verification_reason = verification.reason return working_item

Retriever Tool Contract and Data Shape

The RAG subagent calls search_database(query, expanded_query, limit, wiki_source_filter) in src/backend/tools/retriever_tool.py.

It returns line-oriented evidence in a stable contract:

1. title={title} source={source} content={content} 2. title={title} source={source} content={content} ...

This same line contract is parsed and reused by validation, reranking, subanswer generation, and verification.

Bottlenecks, Trade-offs, and Improvements

Latency Bottlenecks

Bottleneck Why It Costs Time Current Trade-off Potential Improvement
Multiple LLM calls in one run Decomposition + subanswer generation (per subquestion) + initial synthesis + optional refinement path can stack sequentially. Higher answer quality and structure at the expense of wall-clock latency. Use a fast model for decomposition/verification, keep stronger model only for final synthesis; skip optional steps when confidence is high.
Nested parallelism Subquestion thread pool plus per-document validation thread pool can oversubscribe CPU under load. Maximizes throughput for small/medium loads, but can degrade tail latency at high concurrency. Use bounded global worker pools or async queue; cap inner pool to 1 when outer pool is saturated.
Refinement second pass When triggered, pipeline effectively runs retrieval + subquestion processing + synthesis again. Improves recall/coverage for weak first-pass answers. Add stricter refinement gate (confidence + gap score), or partial refinement only on unresolved subquestions.
Context retrieval and formatter passes Multiple retrieval and parse/format passes occur at different stages. Keeps stages decoupled and observable. Cache parsed document objects and reuse across validation/rerank/verification functions.
Frontend timeout set to 10 minutes Long-running requests tie up client wait states and can hide slow regressions. Avoids premature timeout for complex runs. Add progress streaming (SSE/WebSocket), intermediate stage updates, and server-side hard limits per stage.

Accuracy Bottlenecks

Bottleneck Accuracy Risk Current Trade-off Potential Improvement
Decomposition parser fallback If decomposition output is malformed, flow falls back to the original question as one subquestion, reducing decomposition quality. Robustness over optimal decomposition. Schema-constrained LLM output (JSON mode/tool schema) and automatic retry on parse failure.
Heuristic document validation Token-overlap relevance can reject semantically relevant docs and keep lexical false positives. Fast and deterministic filtering. Cross-encoder relevance scoring or hybrid lexical+semantic recheck for borderline documents.
Lightweight reranker scoring Cross-encoder inference can add model download/cold-start overhead. flashrank provides stronger semantic ranking over merged candidates. Keep deterministic non-reranked ordering fallback when reranker is disabled or unavailable.
Verification heuristic Token-overlap grounding check can mark paraphrased but correct answers as ungrounded (false negatives) or vice versa. Cheap guardrail with no extra model call. Use LLM-as-judge with strict citation entailment checks and calibrated thresholds; log disagreement cases.
Final synthesis may compress evidence generate_initial_answer may over-summarize and lose uncertainty nuance despite constraints. Readable final answer for UX. Add structured output (claims + citations + confidence), then render final prose from that structure.
Citation contract is line-based text Parsing depends on strict line format; malformed lines drop evidence silently from downstream stages. Simple interoperability across stages. Replace with typed JSON evidence objects end-to-end; keep line rendering only for logs/UI.

High-Impact, Low-Risk First Improvements

  • Move evidence representation from line text to strict JSON objects between stages.
  • Add stage timing metrics (decomposition, subquestion lane, subquestion pipeline, refinement) and p95 dashboards.
  • Calibrate SUBQUESTION_PIPELINE_MAX_WORKERS and DOCUMENT_VALIDATION_MAX_WORKERS together to avoid oversubscription.
  • Gate refinement with stronger criteria to avoid unnecessary second-pass runs.
  • Add retry-on-parse for decomposition output before fallback to single question.

Graph Runtime Final Answer Responsibility

  • Graph runtime executes typed node stages and synthesizes the final answer in run_synthesize_final_node.
  • Main final answer is generated in backend service layer by generate_initial_answer.
  • UI displays that service-generated output, not intermediate lane text.

Fallback Behavior You Should Know

  • If OPENAI_API_KEY is missing, decomposition/subanswer/initial-answer/refinement-decomposition functions fall back to deterministic local logic.
  • Malformed decomposition output falls back to normalized original question.
  • No parseable retrieved docs yields "No relevant documents found." and likely non-answerable verification.
  • Refinement triggers when initial answer is empty/insufficient, no subanswers, or low answerable ratio (REFINEMENT_MIN_ANSWERABLE_RATIO, default 0.5).
