Run Flow: Runtime, Benchmark, Observability

Current production stage order: decompose -> expand -> search -> rerank -> answer -> synthesize.

UI: App.tsx::handleRun Start: POST /api/agents/run-async Poll: GET /api/agents/run-status/{job_id} Backend: run_runtime_agent() Runner: run_parallel_graph_runner() Response: { main_question, sub_qa, output }

Exact Runtime Chain

Frontend App.tsx::handleRun -> startAgentRun(query) -> poll getAgentRunStatus(job_id) -> update stage rail + decompose/expand/search/rerank/subanswer/final panels Backend routers/agent.py::runtime_agent_run -> services/agent_service.py::run_runtime_agent -> get_vector_store(...) -> search_documents_for_context(...) + build_initial_search_context(...) -> run_parallel_graph_runner(...) -> map_graph_state_to_runtime_response(...) -> RuntimeAgentRunResponse(main_question, sub_qa, output)

  • Benchmark scope emits trace/span/score events for dataset, mode, and question execution.
  • Sampling controls are independent per scope:
  • LANGFUSE_RUNTIME_SAMPLE_RATE, LANGFUSE_BENCHMARK_SAMPLE_RATE.

How Runtime Stages Work

  1. decompose: normalize and dedupe atomic sub-questions.
  2. expand: produce bounded query alternatives with MultiQueryRetriever.
  3. search: retrieve candidates per query, then merge/dedupe deterministically.
  4. rerank: score candidates and trim context depth via top_n.
  5. answer: generate grounded subanswers with citation row linkage.
  6. synthesize: compose final output from subanswers.

Frontend Stage Mapping
