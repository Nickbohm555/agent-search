Run Flow: Runtime, Observability

Current production stage order: decompose -> expand -> search -> rerank -> answer -> synthesize.

Runtime Path

UI: App.tsx::handleRun Start: POST /api/agents/run-async Stream: GET /api/agents/run-events/{job_id} Backend: run_runtime_agent() Runner: run_parallel_graph_runner() Response: { main_question, sub_items, output }

Exact Runtime Chain

Frontend App.tsx::handleRun -> startAgentRun(query) -> subscribeToAgentRunEvents(job_id) -> update stage rail + decompose/expand/search/rerank/subanswer/final panels Backend routers/agent.py::runtime_agent_run -> agent_search.public_api::advanced_rag -> agent_search.runtime.runner::run_runtime_agent -> services/agent_service.py::run_parallel_graph_runner(...) -> map_graph_state_to_runtime_response(...) -> RuntimeAgentRunResponse(main_question, sub_items, output)

Latency / Fallback Hotspots

  • Per-lane search is vector DB/network bound.
  • LLM-heavy nodes: decompose, expand, answer, synthesize.
  • rerank adds another LLM hop per lane when RERANK_PROVIDER=openai.
  • High decomposition count increases lane fanout and queueing under _GRAPH_RUNNER_MAX_WORKERS.
  • Runtime timeout guardrails have been removed, so stage execution is no longer cut short by elapsed-time limits.

Observability Path (Langfuse)

  • Tracing utility: src/backend/utils/langfuse_tracing.py.
  • advanced_rag(..., langfuse_callback=...) or advanced_rag(..., langfuse_settings={...}) enables SDK run tracing.
  • If langfuse_callback is omitted, SDK run tracing is skipped.
  • Sampling control: LANGFUSE_RUNTIME_SAMPLE_RATE.

How Runtime Stages Work

  1. decompose: normalize and dedupe atomic sub-questions.
  2. expand: produce bounded query alternatives with MultiQueryRetriever.
  3. search: retrieve candidates per query, then merge/dedupe deterministically.
  4. rerank: score candidates and trim context depth via top_n.
  5. answer: generate grounded subanswers with citation row linkage.
  6. synthesize: compose final output from subanswers.
