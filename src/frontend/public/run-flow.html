Run Flow: Canonical Graph Runtime

This page documents the current production path only. Canonical stage order: decompose -> expand -> search -> rerank -> answer -> synthesize.

High-Level Path

UI: App.tsx::handleRun Start: POST /api/agents/run-async Poll: GET /api/agents/run-status/{job_id} Backend: run_runtime_agent() Runner: run_parallel_graph_runner() Response: { main_question, sub_qa, output }

Exact Runtime Chain

Frontend App.tsx::handleRun -> startAgentRun(query) -> poll getAgentRunStatus(job_id) -> update stage rail + decompose/expand/search/rerank/subanswer/final panels Backend routers/agent.py::runtime_agent_run -> services/agent_service.py::run_runtime_agent -> get_vector_store(...) -> search_documents_for_context(...) + build_initial_search_context(...) -> run_parallel_graph_runner(...) -> map_graph_state_to_runtime_response(...) -> RuntimeAgentRunResponse(main_question, sub_qa, output)

Pre-graph retrieval is used as decomposition context. The canonical graph lane remains decompose -> expand -> search -> rerank -> answer -> synthesize.

How The Stages Work

  1. decompose: normalize and dedupe atomic sub-questions.
  2. expand: produce bounded query alternatives per sub-question with MultiQueryRetriever.
  3. search: retrieve candidates per expanded query, then merge/dedupe deterministically.
  4. rerank: score candidates with flashrank and trim context to configured top_n.
  5. answer: generate subanswer grounded in reranked evidence and citation rows.
  6. synthesize: compose final output from subanswers while preserving evidence grounding.

Retrieval Fundamentals

  • Vector retrieval uses embedding similarity for initial candidate recall.
  • Cosine similarity ranges from -1..1; higher means closer direction similarity.
  • k_fetch over-fetches candidate rows before reranking.
  • top_n keeps the most relevant reranked rows for answer generation.
  • Merge/dedupe stability: identity is document_id first, fallback is source+content.
  • Citation indices are assigned deterministically from ranked output so [n] markers map stably.

Reranking Fundamentals

  • Vector score and reranker score are different signals.
  • Reranking can move a semantically important row above a raw top-k vector hit.
  • If reranker is disabled/fails, runtime falls back to deterministic original order.
